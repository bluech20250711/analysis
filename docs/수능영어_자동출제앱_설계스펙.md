# 수능 영어영역 모의고사 자동 출제 앱 — 개발 설계 스펙

작성 목적: 클로드 코드(Claude Code)에서 이 문서를 그대로 넘겨 프로젝트를 시작할 수 있도록 하는 전체 스펙.
기존 "이언 ExamBuilder" (React/Vite/TypeScript/Tailwind/Gemini API, Netlify 배포) 스택을 그대로 계승/확장한다.

---

## 0. 프로젝트 개요

**목표**: 한국 수능 영어영역 모의고사 1세트(총 45문항)를 다음 3가지 산출물로 자동 생성하는 웹앱

1. 듣기 1~17번 — 문항(대본) 생성 + 화자별 음성 생성 + 하나의 듣기평가용 MP3로 병합
2. 독해 18~45번 — 문항 유형별 지문/선택지/정답/해설 생성
3. 최종 산출물 — 이언어학원 양식(고등부.hwpx 기준)에 맞춘 HWPX 파일 + PDF 파일 + 듣기 MP3 다운로드

**핵심 제약**
- 실제 수능/모의고사 문항을 표절하지 않고, 유형·구조·난이도만 모사한 **창작 문항**이어야 함
- 문항별 출제 기준은 별도 문서(`수능영어_모의고사_출제_프롬프트.md`)에 정리되어 있으며, 이 문서의 Part1/Part2 표가 Gemini 프롬프트의 핵심 로직이 됨
- 최종 결과물은 이언어학원 실제 시험지(`고등부.hwpx`)와 동일한 레이아웃/표지/문항 번호 스타일이어야 함

## 1. 기술 스택

| 영역 | 선택 | 비고 |
|---|---|---|
| 프론트엔드 | React + Vite + TypeScript + Tailwind | 기존 ExamBuilder와 동일 스택 유지 |
| 문항 생성 AI | Gemini API (`gemini-2.5-pro` 또는 최신 flash 계열) | 기존 프로젝트에서 이미 Tier 1 빌링 세팅 완료 |
| 음성 생성(TTS) | Google Cloud Text-to-Speech (Chirp3-HD / Neural2 voices) | Gemini와 동일 GCP 프로젝트로 통합, API 키/빌링 일원화 |
| 서버리스 백엔드 | Netlify Functions (Node.js) | HWPX/PDF/오디오 병합 처리 (API 키 보관 X, 9절 참고) |
| HWPX 생성 | Node.js + `jszip` (XML 템플릿 주입 방식) | 기존 Python+zipfile 방식의 Node 버전, mimetype은 반드시 STORED(비압축) |
| PDF 생성 | `@react-pdf/renderer` 또는 HTML→PDF(예: Puppeteer 서버리스 함수) | HWPX와 별도 렌더링 파이프라인으로 동일 레이아웃 재현 |
| 오디오 병합/인코딩 | `ffmpeg` (서버리스 함수에 `ffmpeg-static` 바이너리 번들) | 문항별 TTS 클립 + 안내음성 + 신호음 + 무음 간격 삽입 후 하나의 MP3로 병합 |
| 배포 | Netlify | 기존에 겪은 SSR/`server.cjs` 이슈 재발 방지 위해 순수 SPA + Functions 조합으로 설계 (아래 11번 항목 참고) |

## 2. 전체 데이터 흐름

```
[1] 사용자가 옵션 선택
    (회차/난이도/EBS연계여부/특정 학교 스타일 등)
        ↓
[2] Gemini API 호출 (Part1 프롬프트) → 듣기 1-17번 텍스트(대본+선택지+정답+해설) 생성
[3] Gemini API 호출 (Part2 프롬프트) → 독해 18-45번 텍스트(지문+선택지+정답+해설) 생성
        ↓
[4] 듣기 대본을 화자별로 분리 → Google Cloud TTS 호출(문항별 개별 음성 파일 생성)
        ↓
[5] ffmpeg 서버리스 함수: 문항 순서대로 음성 클립 + 안내멘트 + 신호음 + 정적구간 삽입 → 최종 MP3 병합
        ↓
[6] 문항 데이터(JSON) → HWPX 템플릿 XML에 주입 → .hwpx 파일 생성
[6'] 동일 문항 데이터 → PDF 렌더러에 주입 → .pdf 파일 생성
        ↓
[7] 사용자에게 3개 파일 다운로드 제공: 시험지.hwpx / 시험지.pdf / 듣기평가.mp3
     (+ 선택: 정답지 별도 PDF/HWPX)
```

**참고**: [2][3](Gemini)은 브라우저에서 사용자의 키로 직접 호출한다. [4](TTS)와 [5](ffmpeg 병합)는 서버 리소스가 필요하므로 사용자의 TTS 키를 요청에 실어 보내되 서버는 저장하지 않고 그 호출 시점에만 사용한다 (9절 참고).

## 3. 문항 데이터 모델 (TypeScript)

```typescript
// 공통 선택지
interface Choice {
  number: 1 | 2 | 3 | 4 | 5;
  text: string; // 한국어 또는 영어
}

// 듣기 문항 (1-17번)
interface ListeningItem {
  number: number;               // 1~17
  type: string;                 // '목적' | '의견' | '관계' | ... (출제기준표의 유형명)
  instruction: string;          // 실제 시험지 지시문 전체 문장 (예: "다음을 듣고, 남자가 하는 말의 목적으로 가장 적절한 것을 고르시오.")
  speakers: ('M' | 'W' | 'Narrator')[]; // 화자 순서
  script: { speaker: 'M' | 'W' | 'Narrator'; line: string }[]; // 대사 단위 분리 (TTS 개별 호출용)
  scriptKo: string[];            // script와 1:1 대응하는 한국어 해석 (실제 시험지 각주에 원문과 함께 삽입됨)
  choices: Choice[];
  answer: number;
  explanation: string;
  imageRef?: string;            // 4번(그림), 10번(표) 등 이미지 필요 시
  pairGroupId?: string;         // 16-17번처럼 지문 공유 문항 묶음 식별자
}

// 독해 문항 (18-45번)
// 25번(도표)/27-28번(안내문) 공용 표 데이터 형태
interface ReadingTableData {
  caption?: string;
  headers: string[];
  rows: string[][];
}

interface ReadingItem {
  number: number;                // 18~45
  type: string;                  // 출제기준표의 유형명 (목적, 심경, 빈칸추론 등)
  instruction: string;           // 실제 시험지 지시문 전체 문장 (예: "다음 글의 목적으로 가장 적절한 것은?")
  passage: string;                // 지문 본문
  chartData?: ReadingTableData;   // 25번 도표, 27-28번 안내문 표 데이터
  choices: Choice[] | { pairChoices: [Choice[], Choice[]] }; // 40번(요약문 2빈칸)은 별도 구조
  answer: number | string;        // 36-40 등 조합형은 문자열('②') 허용
  explanation: string;
  keyVocab?: { word: string; meaning: string }[];
  imageRef?: string;               // 이미지가 필요한 문항의 설명 문자열 (ListeningItem과 동일 패턴)
  pairGroupId?: string;            // 41-42, 43-45 장문 묶음 식별자
}

interface ExamSet {
  metadata: {
    title: string;              // "2027학년도 O월 모의고사 영어영역"
    academyBranch: '석우관' | '영통관' | '광교관' | string;
    grade: string;               // 고1/고2/고3
    createdAt: string;
  };
  listening: ListeningItem[];    // length 17
  reading: ReadingItem[];        // length 28
}
```

## 4. 문항 생성 (Gemini) 모듈 설계

- 기존 프롬프트 문서(`수능영어_모의고사_출제_프롬프트.md`)의 Part1/Part2 표를 **그대로 시스템 프롬프트에 내장**
- 응답 형식은 자유 텍스트가 아니라 **JSON 강제 출력**으로 변경 (위 `ExamSet` 인터페이스와 1:1 매핑)
  - Gemini에 "You must respond only with valid JSON matching this schema, no markdown, no preamble" 지시 필수
  - JSON 파싱 실패 대비 재시도 로직(최대 2회) 구현
- 분할 생성 전략 (이전 논의대로):
  1. 1차 호출: 듣기 1~17번
  2. 2차 호출: 독해 18~34번
  3. 3차 호출: 독해 35~45번(장문 포함)
  - 각 호출은 이전 호출에서 생성된 소재/어휘를 프롬프트에 함께 전달해 **중복 소재 방지**
- 문항 유형별 파라미터화: 특정 학교 스타일(반송고/동탄고 등), EBS 연계 지문 여부, 난이도(고1~고3)를 UI에서 선택하면 프롬프트 변수로 주입
- Gemini 클라이언트는 사용자가 입력한 API 키(9절)를 인자로 받아 **브라우저에서 직접** 초기화한다 (서버 프록시 불필요)

## 5. 음성 생성(TTS) 모듈 설계

**화자 매핑**
- ListeningItem.script의 각 라인을 화자(M/W/Narrator)별로 분리
- Google Cloud TTS Chirp3-HD 보이스 중 남/여 각 1~2종 고정 배정 (예: 남자=`en-US-Chirp3-HD-Charon`, 여자=`en-US-Chirp3-HD-Kore` 등 — 실제 사용 가능 보이스명은 개발 시 Google Cloud 콘솔에서 최신 목록 확인 필요)
- 16-17번(1인 담화)은 Narrator 전용 보이스 별도 배정

**개별 클립 생성**
- 문항별로 라인 단위 TTS 호출 → 임시 WAV/MP3 클립 개별 저장
- SSML 활용: 문항 간 속도(rate)는 수능 듣기 표준 속도(약 145~155 wpm)에 맞춰 `<prosody rate="...">` 조정

**병합 파이프라인 (ffmpeg)**
- 순서: [듣기 안내멘트] → [문항1 신호음] → [1번 대본] → [정적 15초 표준] → [문항2 신호음] → [2번 대본] → ... → [17번] → [듣기 종료 안내]
- 각 문항 사이 정적 구간 길이는 실제 수능 기준에 맞춰 문항 성격별로 차등 적용 (짧은 문항 12~15초 / 16-17번 공통지문 이후 20초 등 — 정확한 초 단위는 최신 평가원 공식 시간 배분표를 참고해 개발 단계에서 확정)
- 최종 산출물: 단일 MP3 파일 (예: `듣기평가_2027_06.mp3`)

**주의사항**
- Netlify Functions는 기본 실행시간 제한(동기 10초/백그라운드 함수 최대 15분)이 있으므로, ffmpeg 병합처럼 시간이 걸리는 작업은 **Background Function**으로 분리하고 프론트에서는 폴링(polling) 또는 웹훅으로 완료 여부 확인하는 구조 필요
- TTS 호출 자체(개별 클립 생성)는 사용자의 TTS 키로 브라우저에서 직접 호출 가능하면 그렇게 하고, CORS 등 제약으로 어려우면 요청 시점에만 키를 서버로 전달해 즉시 사용 후 폐기하는 방식 사용(9절 참고)

## 6. HWPX 출력 모듈 설계

**분석된 템플릿 구조 (`고등부.hwpx` 기준)**

```
mimetype                    (STORED, "application/hwp+zip")
version.xml
Contents/header.xml         (폰트/스타일 정의)
Contents/masterpage0.xml    (듣기 페이지 마스터)
Contents/masterpage1.xml    (독해 페이지 마스터)
Contents/section0.xml        (실제 문항 본문 — 문항 반복 구조)
Contents/content.hpf
BinData/image1~4.jpg         (그림/표/도표 이미지)
Preview/PrvText.txt, PrvImage.png
META-INF/container.xml, container.rdf, manifest.xml
settings.xml
```

**생성 전략**
1. `section0.xml`에서 문항 1개의 XML 블록(문단+표+이미지참조)을 "템플릿 조각"으로 추출
2. 문항 데이터(JSON)를 순회하며 템플릿 조각을 복제 → 번호/지문/선택지 텍스트만 치환
3. 이미지가 필요한 문항(4, 10, 25번 등)은 `BinData/`에 새 이미지 추가 + `content.hpf`의 매니페스트에 참조 등록
4. 완성된 XML들을 `jszip`으로 재압축
   - **`mimetype` 파일은 반드시 압축하지 않고(STORED) 맨 처음 항목으로 저장**
   - 출력 파일명은 **ASCII 문자만 사용** (한글 파일명은 다운로드 시 깨짐 발생 이력 있음 — 예: `exam_2027_06_set1.hwpx`로 저장 후 사용자에게 노출되는 표시명만 한글로)
5. 표지 텍스트(회차, 학원 지점명 등)는 masterpage XML의 해당 텍스트 노드만 치환

## 7. PDF 출력 모듈 설계

- HWPX와 완전히 별도 파이프라인으로 동일 레이아웃을 재현 (HWPX→PDF 직접 변환은 안정성이 낮아 권장하지 않음)
- 방식 A: `@react-pdf/renderer`로 시험지 컴포넌트를 React로 재작성 (2단 편집, 원문자 선택지, 표지 포함)
- 방식 B: 동일한 HTML/CSS 템플릿을 만들어 Puppeteer 서버리스 함수로 PDF 변환
- 두 방식 다 이언어학원 표지 양식(회차/지점/학년 표기)을 그대로 반영
- 정답지(해설 포함) PDF는 별도 문서로 생성 옵션 제공

## 9. API 키 관리 방식 — 사용자 개별 입력 (BYOK: Bring Your Own Key)

**중요한 설계 변경**: 이 앱은 로그인 기능이 없는 멀티유저 도구(이언어학원 여러 지점 선생님들이 공용으로 사용)이므로, API 키를 서버(.env)에 고정하지 않고 **각 사용자가 자신의 Gemini API 키와 Google Cloud TTS 키를 직접 입력**하도록 설계한다.

**동작 방식**
1. 앱 최초 진입 시 "API 키 설정" 화면(또는 설정 모달)에서 두 키를 입력받음
   - Gemini API 키
   - Google Cloud TTS API 키
2. 입력된 키는 **브라우저의 `localStorage`에만 저장** (서버로 전송/저장하지 않음)
3. Gemini/TTS API 호출은 **브라우저에서 사용자의 키로 직접 호출** — 더 이상 서버(Netlify Functions)가 키를 대신 보관하거나 프록시할 필요 없음
   - 단, ffmpeg 오디오 병합처럼 서버 리소스가 필요한 작업(Background Function)에는 사용자의 TTS 키를 요청 body에 함께 전달해서 서버가 해당 호출 시점에만 일회성으로 사용 (저장하지 않고 즉시 폐기)
4. 키가 없거나 유효하지 않을 경우: 문항 생성/음성 생성 버튼 클릭 시 명확한 안내 메시지와 함께 설정 화면으로 유도
5. 기기를 바꾸거나 브라우저 데이터를 지우면 키를 다시 입력해야 함 — UI에 이 점을 안내 문구로 명시

**보안 관련 유의사항 (개발 시 클로드 코드에 반드시 전달)**
- `localStorage`에 저장된 키는 XSS 공격에 취약할 수 있으므로, 이 앱에는 사용자가 입력한 임의의 HTML/스크립트가 렌더링되는 부분이 없도록 주의 (생성된 지문 텍스트 등을 `dangerouslySetInnerHTML`로 렌더링하지 않기)
- 키 입력 필드는 `type="password"`로 마스킹 처리
- 콘솔 로그, 에러 메시지, 네트워크 요청 로그 어디에도 키 값이 그대로 노출되지 않도록 처리

**향후 확장 여지**: 나중에 로그인 기능을 추가하게 되면, 키를 서버 DB에 사용자별로 암호화 저장하는 방식으로 전환 가능 (지금 단계에서는 범위 밖)

**남은 서버 환경변수 (없음 또는 최소화)**
```
# 현재 설계에서는 서버가 보관하는 공용 API 키가 없음
# 추후 관리자 전용 기능(예: 사용량 모니터링) 추가 시에만 별도 환경변수 필요
```

## 11. 배포 시 유의사항 (기존 이슈 반영)

- 이전 ExamBuilder에서 겪은 Remix/Vite SSR + `dist/server.cjs` 이슈 재발 방지를 위해, 이 프로젝트는 **순수 SPA(정적 빌드) + Netlify Functions** 구조로 설계 (SSR 불필요)
- `netlify.toml`에 functions 디렉터리, background function 설정, redirects(SPA fallback) 명시 필요
- ffmpeg 바이너리는 서버리스 함수 배포 용량 제한(보통 50MB)을 고려해 `ffmpeg-static` 등 경량 패키지 사용 검토, 초과 시 별도 오디오 처리 전용 서비스(예: 별도 컨테이너/Cloud Run) 분리도 대안으로 고려
